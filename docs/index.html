비밀번호 해싱

기존

비밀번호+솔트를 SHA-256으로 2회 반복 해시해 저장. 평문 저장은 피했고 솔트로 동일 비밀번호의 같은 해시는 방지했으나, SHA-256은 "빠르라고" 설계돼 GPU로 초당 수십억 번 시도 가능. 2회 반복은 공격 비용을 사실상 늘리지 못해 DB 유출 시 약한 비밀번호는 단시간에 역산된다.

개선

PBKDF2(SHA-256, 반복 10만 회, 32바이트 출력)로 교체. 해시 1회 비용이 의도적으로 비싸져 무차별 대입 비용이 수만 배 증가. .NET 표준 라이브러리(Rfc2898DeriveBytes.Pbkdf2) 내장이라 외부 의존성 없고 NIST 승인 알고리즘.

대안과 선택 이유

bcrypt/Argon2가 더 강력하지만(특히 Argon2는 메모리까지 소모) 둘 다 외부 NuGet 의존이 생긴다. 표준 라이브러리만으로 충분해 PBKDF2 선택. 트레이드오프: 기존 해시가 전부 무효화됨 — 운영 전 단계라 점진적 재해시 대신 단순 무효화(계정 재생성)를 택했다.

시크릿 관리와 JWT 키 인코딩

기존

JWT secret과 DB 비밀번호가 appsettings.json에 평문 커밋. 토큰 발급은 키를 UTF-8로, 검증은 ASCII로 인코딩. 저장소 접근만 있으면 토큰 위조·DB 접속 가능하고, 인코딩 불일치는 non-ASCII가 secret에 들어가는 순간 디버깅 지옥을 만든다.

개선

secret을 User Secrets(개발)/환경변수(배포)로 이동, 인코딩을 UTF-8로 통일. 코드와 자격증명 분리, .NET 구성 계층이 표준 지원해 코드 변경 없이 환경별 주입 가능.

대안과 선택 이유

Azure Key Vault 같은 시크릿 매니저는 현재 규모엔 인프라 과잉. 가장 가벼운 표준 방법을 선택.

잔여 과제: 이미 git 히스토리에 남은 secret은 로테이션 필요.

세션 인증 구조 가장 큰 변경

기존

커스텀 미들웨어가 모든 요청 body를 버퍼링→JSON 파싱해 userId/authToken을 추출, Redis 세션과 대조. 어떤 경로가 어떤 인증을 쓰는지는 미들웨어 안의 하드코딩 경로 목록 2개가 결정.

  • body를 미들웨어와 모델바인딩이 두 번 파싱
  • body 없는 요청은 인증 불가 — GET /Character/owned는 사실상 호출 불가
  • 자격증명이 body에 있어 요청 로깅에 유출 위험
  • 새 컨트롤러마다 경로 목록 수정 필요, 잊으면 깨지거나 인증 누락
  • /login(소문자) 거부, /Accountxyz 우회 같은 매칭 버그
개선

2단계로 진행: ① 자격증명을 X-User-Id/X-Auth-Token 헤더로 이동(body 파싱 삭제) ② 미들웨어를 ASP.NET Core 표준 AuthenticationHandler("Session" 스킴)로 교체. 각 컨트롤러가 [Authorize]/[AllowAnonymous]로 정책을 선언하고, FallbackPolicy가 선언 없는 엔드포인트에 자동으로 인증 요구.

  • 경로 목록 제거 → 정책이 엔드포인트 옆에 보임
  • 선언을 잊어도 기본이 거부(deny-by-default)
  • GET 인증 가능, body 이중 파싱 제거, 자격증명 로깅 차단
대안과 선택 이유

(a) 미들웨어 유지+목록 정리 → "중앙 목록 관리" 근본 문제 잔존, 기각. (b) JWT 하나로 통일 → JWT는 즉시 무효화 불가(로그아웃·강제차단), 게임 서버는 세션 제어 필요해 기각. (c) Bearer 하나에 둘 다 싣기 → 스킴 판별 모호, 기각. (d) ASP.NET Core Identity 풀 채택 → 회원관리 전체가 따라와 과잉, 기각.

선택: 커스텀 인증을 프레임워크 확장점에 맞춰 넣는 것이 코드량은 비슷하면서 표준 생태계 전부를 얻는 방법.

부수 효과: 인증 실패가 400 → 401 + 기존 {"errorCode":1006} 형태 유지 (클라이언트 수정 필요).

신원 처리와 API 계약

기존

UserController는 body의 request.UserId를 신뢰, CharacterController는 미들웨어 검증값 사용 — 같은 의미의 값을 다른 출처에서 읽음. 응답은 User EF 엔티티를 그대로 직렬화. 유저 없음/닉네임 중복엔 KeyNotFoundException을 던져 클라이언트가 500을 받음. 신원 이원화는 IDOR(타인 데이터 접근)의 씨앗.

개선

검증된 userId는 HttpContext.GetAuthenticatedUserId() 단일 출처로 통일, DTO에서 UserId 제거. 응답은 UserInfo DTO. 예외는 GameException + 전용 코드(UserNotFound=1009, DuplicateNickname=1010)로 400 응답.

대안과 선택 이유

신원을 Claims에서 직접 읽으면 매번 문자열→long 파싱 필요 → 핸들러가 HttpContext.Items에 넣고 확장 메서드로 읽는 방식(변환 로직 한곳). DTO 매핑은 AutoMapper 대신 수동 — 엔티티 1~2개 규모에 매퍼는 과잉.

동시 요청 직렬화 (유저 락)

기존

분산락이 JWT principal의 account 기준으로만 걸림. 세션 인증 경로(/Character/gacha 등)는 JWT principal이 없어 락이 아예 안 걸림 → 동시 가챠 요청으로 재화 이중 사용 가능.

개선

검증된 userId 기준 lock:user:{id} 락을 우선 적용, JWT 경로는 account 락 유지. 같은 유저 요청이 직렬화되어 read-decide-write 구간이 보호됨.

대안과 선택 이유

DB 비관적 락이나 낙관적 동시성도 가능하지만, DB 락은 트랜잭션 범위만 보호하고 Redis 캐시 정합까지는 못 지킨다. 이미 Redis 분산락 인프라가 있고 "요청 단위 직렬화"가 캐시 연산까지 한 번에 보호하므로 선택.

가챠 캐시/DB 정합성 머니 버그

기존 (실버그)

재화 잔액·보유 캐릭터 판정을 Redis 캐시 기준으로 하고 저장 후 캐시에 다시 써넣음(write-through). User가 캐시 히트로 오면 차감 결과가 dirty로 표시되지 않아 Redis에 옛 잔액이 24시간 잔존. 다음 가챠가 stale 잔액으로 계산한 값을 덮어써서 — 1000→700→(캐시 700 기준)400 — 쓴 재화가 사실상 환불. DB 커밋 후 Redis 쓰기 실패 시 stale 데이터가 TTL까지 잔존하는 원자성 문제도.

개선

재화·소유 판정은 DB 직접 조회(5번 유저 락이 직렬화 보장), 저장 성공 후 캐시는 갱신하지 않고 무효화(delete). 무효화 실패는 경고 로그만 남기고 요청은 성공. 돈 계산의 진실 출처가 DB 하나로 고정, 캐시 실패의 최악 결과가 "캐시 미스"로 수렴.

대안과 선택 이유

(a) write-through 유지+user키 갱신 → 원자성 구조 문제 잔존. (b) 트랜잭셔널 아웃박스 → 현 규모에 과잉. (c) 캐시 전면 제거 → 조회 성능 손해. 선택: cache-aside + 쓰기 후 무효화 — 업계 표준이고 실패 시 안전한 쪽으로 기운다.

닉네임 검증과 중복 race

기존

길이/형식 제한 없음(1000자 닉네임 가능, DB varchar(16)이라 DB 에러→500). 중복 검사는 조회→저장 사이 다른 요청이 끼어들 수 있고, unique index 위반이 500으로 노출.

개선

서비스에서 2~16자 + 앞뒤 공백 금지 검증. race에서 진 요청의 DbUpdateException(uk_nickname 위반)을 DuplicateNickname으로 변환. 사전 검사(친절한 에러) + DB unique index(최후 안전망) + 예외 변환(클라 친화) 3중 구조.

대안과 선택 이유

DB 제약만 믿으면 정상 케이스까지 예외 흐름이 됨. 둘 다 유지가 비용 대비 명확. 검증 위치는 DataAnnotations 대신 서비스 — [ApiController] 자동 400은 GameResponse 형태가 아니라 에러 계약과 어긋나기 때문.

운영 보안: 로깅과 세션 수명

기존

세션 토큰을 Info 로그에 평문 기록, EF의 EnableSensitiveDataLogging+SQL 콘솔 출력이 모든 환경에서 활성. 세션은 24시간 TTL 외 무효화 수단 없음. 로그 접근=세션 탈취, 프로덕션 로그에 쿼리 파라미터(이메일·재화) 잔존, 탈취 토큰 차단 불가.

개선

대안과 선택 이유

로그 마스킹 필터로 가리는 방법도 있지만 애초에 안 찍는 것이 가장 단순·확실. 로그아웃은 클라 측 토큰 폐기만으론 서버 무효화가 안 돼(탈취 토큰이 TTL까지 유효) 서버 삭제 방식 선택.

  • 현재 L1(IRequestScopedLocalCache)은 AddScoped = 요청 1개 수명. 그래서 오늘은 프로세스에 오래 남는 stale이 없고, 공유 진실은 Redis 하나뿐 → RemoveAsync가 모든 노드에 즉시 반영된다.
  • HybridCache의 L1은 프로세스 수명. plain Redis IDistributedCache엔 backplane이 없어 RemoveAsync가 다른 노드의 L1까지 지우지 못한다 → 가챠 직후 다른 노드로 LB된 조회가 최대 LocalCacheExpiration 동안 신규 캐릭터를 누락(= 현재보다 약한 정합성, P4 E2E를 깨뜨림).
  • 게다가 이 코드에선 player:characters 키를 요청당 1회만 읽는다 → 프로세스 L1의 이득이 사실상 0.

선택: 이미 있는 IDistributedCacheAdapter 위에 얇은 cache-aside 래퍼(~15줄). "Redis가 공유 진실, 프로세스 메모리에 안 남김" 모델과 1:1로 맞고 새 의존성이 0. HybridCache는 나중에 여러 곳에서 자주 읽는 공용 핫키가 생기면 그때 도입한다.

Phase 1 — 돈을 DB 조건부 UPDATE로 보호 정확성·최우선 구현 완료
P1

인메모리 차감 폐기, 원자적 차감 + 캐릭터 INSERT를 한 트랜잭션으로 구현 완료

RTWWebServer/Services/CharacterGachaService.cs · Data/Repositories/UserRepository.cs
문제

현재는 user.PremiumCurrency -= actualCost; userRepository.Update(user)로 인메모리 계산 후 저장. 차감의 정확성이 사실상 Redis 분산락(30s TTL) 하나에만 기댄다. 락이 풀리거나 페일오버하면 더블스펜드 가능.

방침

IUserRepository.TryDeductPremiumCurrencyAsync(조건부 UPDATE)를 추가하고, 명시적 트랜잭션으로 차감 + 캐릭터 INSERT를 한 단위로 묶는다.

  • 순서: ① 소유 캐릭터 조회→뽑기 ID 선정→actualCost 계산 ② BeginTransactionAsync ③ TryDeduct→false면 롤백+InsufficientCurrency ④ 캐릭터 AddAsync+SaveChangesAsync ⑤ 커밋
  • ExecuteUpdateAsync는 즉시 실행이라, 캐릭터 INSERT와 원자적으로 묶으려면 같은 컨텍스트의 트랜잭션에 참여시켜야 한다.
주의 — 구현 시 반드시
  • 커밋 후 잔액 재조회는 트래킹된 User를 잡으면 안 된다. ExecuteUpdateAsync는 체인지 트래커를 우회하므로, 차감 전 user를 로드해 두면 FindAsync(=GetByIdAsync)가 옛 잔액을 반환한다. → 차감 전 user를 로드하지 말고, 커밋 후 AsNoTracking으로 새로 읽어 응답을 채운다.
  • 중복 캐릭터도 DB가 막아준다. (UserId, CharacterMasterId)에 유니크 인덱스가 있어(GameDbContext.cs:106) 동시 가챠가 같은 캐릭터를 뽑으면 두 번째 INSERT가 위반 → 같은 트랜잭션 통째 롤백(차감 포함) → 돈 손실 없음. 단 DbUpdateException(DuplicateKeyEntry)을 잡아 conflict 에러로 변환해 500을 막는다.
구현 결과 · d53b81f
  • 계획대로 TryDeductPremiumCurrencyAsync(조건부 UPDATE) + 명시적 트랜잭션으로 차감·INSERT 원자화. 순서 ①~⑤ 그대로 적용.
  • 중복 캐릭터 DbUpdateException(DuplicateKeyEntry) → 트랜잭션 롤백 + 신규 에러코드 DuplicateCharacter(1013). 응답 잔액은 커밋 후 GetByIdAsNoTrackingAsync로 재조회.
  • 단위 테스트를 새 계약(TryDeduct/GetByIdAsNoTracking mock)에 맞춰 갱신 — 63개 전부 통과, 빌드 경고 0. InMemory는 트랜잭션 미지원이라 TransactionIgnoredWarning을 무시 처리.
  • 실제 차감 원자성(트래커-stale·차감+INSERT)의 검증은 아직 P4의 DB 통합 테스트에 달려 있다 — mock으로는 못 잡는 부분이라 보류 중.
