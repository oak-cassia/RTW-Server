# RTW 웹서버 개선 보고서 (2026-06-11 ~ 06-12)

13개 커밋(`3bf02b1`~`27718d2`)을 주제별로 묶어 정리했다.
각 항목은 **기존 동작 → 개선 동작 → 대안과 선택 이유** 순서로 기술한다.

---

## 1. 비밀번호 해싱

### 기존 동작
비밀번호+솔트를 SHA-256으로 2회 반복 해시해 저장.

- **효과**: 평문 저장은 피했고, 솔트 덕에 동일 비밀번호가 같은 해시가 되는 것은 방지.
- **단점**: SHA-256은 "빠르라고" 설계된 해시라 GPU로 초당 수십억 번 시도가 가능. 2회 반복은 공격 비용을 사실상 늘리지 못해, DB가 유출되면 약한 비밀번호는 단시간에 역산됨.

### 개선 동작
PBKDF2(SHA-256, 반복 10만 회, 32바이트 출력)로 교체.

- **효과**: 해시 1회 계산 비용이 의도적으로 비싸져 무차별 대입 비용이 수만 배 증가.
- **장점**: .NET 표준 라이브러리(`Rfc2898DeriveBytes.Pbkdf2`) 내장이라 외부 의존성 없음. NIST 승인 알고리즘.

### 대안과 선택 이유
bcrypt/Argon2가 더 강력하고(특히 Argon2는 메모리까지 소모시켜 GPU 공격에 더 강함) 일반적 권장이지만, 둘 다 외부 NuGet 패키지 의존이 생긴다. 표준 라이브러리만으로 충분한 수준을 확보할 수 있어 PBKDF2를 선택.

**트레이드오프**: 기존 해시가 전부 무효화됨. "로그인 성공 시 점진적 재해시" 마이그레이션 대안도 있었지만, 아직 운영 전 단계라 단순 무효화(계정 재생성)를 택했다.

## 2. 시크릿 관리와 JWT 키 인코딩

### 기존 동작
JWT secret과 DB 비밀번호가 `appsettings.json`에 평문으로 커밋됨. 토큰 발급은 키를 UTF-8로, 검증은 ASCII로 인코딩.

- **단점**: 저장소 접근 권한만 있으면 누구나 토큰을 위조하고 DB에 접속 가능. 인코딩 불일치는 non-ASCII 문자가 secret에 들어가는 순간 "발급한 토큰이 검증에 실패"하는 디버깅 지옥을 만듦.

### 개선 동작
secret을 User Secrets(개발)/환경변수(배포)로 이동, 인코딩을 UTF-8로 통일.

- **장점**: 코드와 자격증명의 분리. .NET 구성 계층이 표준 지원하므로 코드 변경 없이 환경별 주입 가능.

### 대안과 선택 이유
Azure Key Vault 같은 시크릿 매니저는 운영 단계에서 고려할 가치가 있지만 현재 규모엔 인프라 과잉. 가장 가벼운 표준 방법을 선택.

> **잔여 과제**: 이미 git 히스토리에 남은 secret은 로테이션 필요.

## 3. 세션 인증 구조 (가장 큰 변경)

### 기존 동작
커스텀 미들웨어가 **모든 요청의 body를 버퍼링 → JSON 파싱**해서 `userId`/`authToken` 필드를 추출하고 Redis 세션과 대조. 어떤 경로가 어떤 인증을 쓰는지는 미들웨어 안의 **하드코딩 경로 목록 2개**(`EXCLUDED_PATHS`, `JWT_AUTHENTICATED_PATHS`)가 결정.

- **효과**: JWT(계정) → 세션(게임 유저)의 2단계 인증 흐름 자체는 동작.
- **단점**:
  - body를 미들웨어와 모델바인딩이 **두 번 파싱** (성능 + 복잡도)
  - **body 없는 요청은 인증 불가** — `GET /Character/owned`는 사실상 호출 불가능한 엔드포인트였음
  - 자격증명이 body에 있어 요청 로깅에 유출될 위험
  - 새 컨트롤러마다 미들웨어의 경로 목록을 수정해야 하고, 잊으면 엔드포인트가 깨지거나 인증이 빠짐
  - `/login`(소문자) 거부, `/Accountxyz` 인증 우회 같은 경로 매칭 버그

### 개선 동작
2단계로 진행:

1. 자격증명을 `X-User-Id`/`X-Auth-Token` **헤더**로 이동 (body 파싱 전부 삭제)
2. 미들웨어 자체를 ASP.NET Core 표준 **`AuthenticationHandler`**("Session" 스킴)로 교체. 각 컨트롤러가 `[Authorize(AuthenticationSchemes=...)]` / `[AllowAnonymous]`로 정책을 **선언**하고, FallbackPolicy가 선언 없는 엔드포인트에 자동으로 인증을 요구.

- **효과**: 인증 흐름(JWT → 세션 발급)은 그대로, 전달 방식과 적용 메커니즘만 표준화.
- **장점**:
  - 경로 목록이 사라지고 정책이 **엔드포인트 옆에** 보임
  - 새 컨트롤러에서 선언을 잊어도 **기본이 거부**(deny-by-default)라 보안 사고로 이어지지 않음
  - GET 요청 인증 가능, body 이중 파싱 제거, 자격증명 로깅 유출 경로 차단
  - 프레임워크의 `[Authorize]`/정책/역할 생태계를 그대로 활용 가능

### 대안과 선택 이유
| 대안 | 판단 |
|---|---|
| (a) 미들웨어 유지 + 경로 목록만 정리 | 매칭 버그는 고쳐지지만 "중앙 목록을 계속 관리해야 한다"는 근본 문제가 남음 → 기각 |
| (b) 세션 토큰을 버리고 JWT 하나로 통일 | 스킴이 하나로 단순해지지만, JWT는 서버가 **즉시 무효화할 수 없음**(로그아웃·강제 차단 불가, 블랙리스트를 만들면 결국 Redis 조회가 부활) → 게임 서버는 세션 제어가 필요하므로 기각 |
| (c) `Authorization: Bearer` 하나에 JWT와 세션 토큰을 모두 싣고 형태로 구분 | 표준 헤더라는 장점은 있지만 스킴 판별 로직이 모호해짐 → 기각 |
| (d) ASP.NET Core Identity 풀 채택 | 회원 관리 기능 전체가 따라와 게임 서버엔 과잉 → 기각 |

**선택**: 커스텀 인증을 프레임워크의 확장점(AuthenticationHandler)에 맞춰 넣는 것이 코드량은 비슷하면서 표준 생태계 전부를 얻는 방법.

부수 효과: 인증 실패가 400 → **401** + 기존 `{"errorCode":1006}` 형태 유지 (클라이언트 수정 필요).

## 4. 신원 처리와 API 계약

### 기존 동작
UserController는 body의 `request.UserId`를 신뢰, CharacterController는 미들웨어가 검증한 값을 사용 — 같은 의미의 값을 다른 출처에서 읽음. 응답은 `User` **EF 엔티티를 그대로 직렬화**. 유저 없음/닉네임 중복엔 `KeyNotFoundException`을 던져 클라이언트가 **500 서버 오류**를 받음.

- **단점**: "클라이언트가 주장하는 신원"과 "서버가 검증한 신원"의 이원화는 IDOR(타인 데이터 접근) 사고의 씨앗. 엔티티 노출은 `AccountId` 등 내부 컬럼이 새고, 나중에 추가될 컬럼도 자동으로 노출됨. 닉네임 중복(정상적 입력 실수)과 서버 장애를 클라이언트가 구분 불가.

### 개선 동작
검증된 userId는 `HttpContext.GetAuthenticatedUserId()` **단일 출처**로 통일하고 DTO에서 `UserId` 제거. 응답은 `UserInfo` DTO. 예외는 `GameException` + 전용 코드(`UserNotFound=1009`, `DuplicateNickname=1010`)로 400 응답.

- **장점**: 신원 위조 여지 제거, DB 스키마와 API 계약의 분리, 클라이언트가 에러 코드로 분기 가능.

### 대안과 선택 이유
신원은 Claims에서 직접 읽는 방법도 있지만 매번 문자열→long 파싱이 필요해, 핸들러가 검증 시점에 `HttpContext.Items`에 넣고 확장 메서드로 읽는 방식을 택했다(변환 로직이 한곳). DTO 매핑은 AutoMapper 도입 대신 수동 매핑 — 엔티티 1~2개 규모에 매퍼 라이브러리는 과잉.

## 5. 동시 요청 직렬화 (유저 락)

### 기존 동작
분산락이 JWT principal의 account 기준으로만 걸림.

- **단점**: 세션 인증 경로(`/Character/gacha` 등)는 JWT principal이 없어 **락이 아예 안 걸림** → 동시 가챠 요청으로 재화 이중 사용 가능.

### 개선 동작
검증된 userId 기준 `lock:user:{id}` 락을 우선 적용, JWT 경로는 account 락 유지.

- **장점**: 같은 유저의 요청이 직렬화되어 read-decide-write 구간이 보호됨.

### 대안과 선택 이유
DB 비관적 락(`SELECT ... FOR UPDATE`)이나 낙관적 동시성(version 컬럼)도 가능하지만, DB 락은 트랜잭션 범위만 보호하고 Redis 캐시와의 정합까지는 못 지킨다. 이미 Redis 분산락 인프라가 있고, "요청 단위 직렬화"가 캐시 연산까지 한 번에 보호하므로 이를 선택.

## 6. 가챠 캐시/DB 정합성 (이번에 발견한 머니 버그)

### 기존 동작
재화 잔액·보유 캐릭터 판정을 **Redis 캐시 기준**으로 하고, 저장 후 캐시에 다시 써넣음(write-through).

- **단점 (실버그)**: User가 캐시 히트로 오면 차감 결과가 dirty로 표시되지 않아 **Redis에 옛 잔액이 24시간 잔존**. 다음 가챠가 그 stale 잔액으로 계산한 값을 DB에 덮어써서 — 예: 1000 → 700 → (캐시 700 기준) 400 → 실제로는 100이어야 하는데 400 — **쓴 재화가 사실상 환불**됨. 또 DB 커밋 후 Redis 쓰기가 실패하면 stale 데이터가 TTL까지 남는 원자성 문제.

### 개선 동작
재화·소유 **판정은 DB 직접 조회**(5번의 유저 락이 직렬화를 보장하므로 안전), 저장 성공 후엔 캐시를 갱신하지 않고 **무효화(delete)**. 무효화 실패는 경고 로그만 남기고 요청은 성공 처리.

- **장점**: 돈 계산의 진실 출처가 DB 하나로 고정. 캐시 실패의 최악 결과가 "캐시 미스(다음 조회 때 DB에서 재적재)"로 수렴 — 가장 안전한 실패 모드. 조회 API(`/Character/owned`)의 캐시 이점은 유지.

### 대안과 선택 이유
| 대안 | 판단 |
|---|---|
| (a) write-through 유지 + user 키도 갱신 | 버그는 고쳐지나 "DB 커밋과 Redis 쓰기가 원자적이지 않다"는 구조 문제가 남음 |
| (b) 트랜잭셔널 아웃박스 등 이벤트 기반 캐시 동기화 | 정석이지만 현 규모에 과잉 |
| (c) 캐시 전면 제거 | 단순하지만 조회 성능 손해 |

**선택**: cache-aside + 쓰기 후 무효화. 업계 표준 패턴이고, 실패 시 안전한 쪽(캐시 미스)으로 기울어짐.

## 7. 닉네임 검증과 중복 race

### 기존 동작
길이/형식 제한 없음(1000자 닉네임 가능, DB `varchar(16)`이라 DB 에러 → 500). 중복 검사는 조회→저장 사이에 다른 요청이 끼어들 수 있고, 그 경우 unique index 위반이 500으로 노출.

### 개선 동작
서비스에서 2~16자 + 앞뒤 공백 금지 검증. race에서 진 요청의 `DbUpdateException`(uk_nickname 위반)을 `DuplicateNickname`으로 변환.

- **장점**: 사전 검사는 대부분의 경우 빠르고 친절한 에러를 주고, DB unique index가 최후의 안전망, 예외 변환이 그 안전망을 클라이언트 친화적으로 만듦 — 3중 구조.

### 대안과 선택 이유
사전 검사를 없애고 DB 제약만 믿는 방법도 있지만(코드 단순), 정상 케이스까지 예외 흐름으로 처리하게 된다. 둘 다 유지가 비용 대비 명확하다. 검증 위치는 DataAnnotations(DTO) 대신 서비스 — `[ApiController]`의 자동 400은 `GameResponse` 형태가 아니어서 프로젝트 에러 계약과 어긋나기 때문.

## 8. 운영 보안: 로깅과 세션 수명

### 기존 동작
세션 토큰을 Info 로그에 평문 기록, EF의 `EnableSensitiveDataLogging`+SQL 콘솔 출력이 모든 환경에서 활성. 세션은 24시간 TTL 외에 무효화 수단 없음.

- **단점**: 로그 접근 = 세션 탈취. 프로덕션 로그에 쿼리 파라미터(이메일, 재화)가 남음. 탈취된 토큰을 차단할 방법이 없음.

### 개선 동작
토큰 로깅 제거 + 구조적 로깅 전환, EF 민감 로깅은 `IsDevelopment` 한정, `POST /User/logout`으로 Redis 세션 즉시 삭제.

- **장점**: 자격증명이 로그에 안 남고, 유출 의심 시 서버가 세션을 끊을 수 있음.

### 대안과 선택 이유
로그 마스킹 필터(Serilog destructuring 등)로 토큰을 가리는 방법도 있지만, **애초에 안 찍는 것**이 가장 단순하고 확실하다. 로그아웃은 클라이언트 측 토큰 폐기만으로는 서버 측 무효화가 안 되므로(탈취 토큰이 TTL까지 유효) 서버 삭제 방식을 택했다.

- 단위 테스트 62개 전체 통과 (인증 핸들러 테스트 신규 6개 포함)
- 로컬 MySQL/Redis로 실서버 E2E:
  - 게스트 생성 → JWT → 세션 발급 → 닉네임/가챠/조회 → 로그아웃 → 폐기 토큰 401
  - 부정 케이스(헤더 누락·위조 토큰·JWT 없음) 모두 401 + 에러코드 확인
